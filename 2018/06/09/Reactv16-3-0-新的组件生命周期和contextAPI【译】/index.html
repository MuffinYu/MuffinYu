官方的 Context API

长期以来，React 已经提供过一个实验性的 context API。它是个非常强大的工具，他的用法却很让人沮丧，因为这个 API 与生俱来的一些问题，同时，我们也一直打算用一个更好的 API 来替代这个实验性的。

16.3 版本中将引入一个新的 context API，他将更加的高效，同时支持静态类型检查和深层次更新。

注意
原来的 context API 在 React 16.x 的版本种仍然可以使用，所以你有充足的时间来迁移。

createRef App

以前，React 提供了两种方式管理 refs：传统字符串 ref API 和 callback API。尽管字符串 ref API 是更便捷的方式，他也有一些缺点，因此我们官方也推荐使用 callback 的方式替代字符串方式。

16.3 版本增加了一种新的方式管理 refs，它提供了一种便利的字符串 ref，但是没有那些缺陷。

注意
callback refs 的形式仍然会被支持，作为新的 createRef API 的补充。
你不必替换你组件里的 callback refs。他们更加的灵活，所以他们仍然会被保留作为高级特性。

组件生命周期更改

React 的类组件 API 已经好几年没有大的改动了。然而，随着我们对更多高级特性的支持（比如，错误边界和即将使用的异步渲染模式（async rendering mode），我们不断扩展这个框架，这其中有些内容是以前没有设计好的。

比如，使用目前的 API，太容易因为一些不重要的逻辑而阻塞初始化渲染。某种意义上，这是因为有太多的方式去完成一些特定的任务，到底哪种方式是最优方式反而不太清晰。我们注意到错误处理的操作中断行为经常会被忽略，可能会造成内存泄漏的问题（会影响接下来的异步渲染模式）。现阶段的类组件API也使一些其他的工作复杂化了，比如我们在React编译器原型上的工作。

某些组件生命周期函数加剧了这些问题（componentWillMount, componentWillReceiveProps，和componentWillUpdate）。同时这些生命周期函数也在React社区造成了很多困惑。基于以上原因，我们决定放弃这些方法，采用更好的方法替代。

我们承认这些改变会影响很多已经存在的组件。所以这条迁移之路会尽可能的平缓，同时我们也会提供逃生舱。（在Facebook, 我们也维护着超过 50,000 个 React 组件。我们也依赖渐进的发布周期）

注意
反对警告会在将来的 16.x 版本中使用，但是遗承生命周期会一直到17版本。
甚至在17版本中，仍然可以使用他们，但是他们会被加上 “UNSAFE_” 的前缀，表示他们可能会造成一些问题。我们准备了一个自动执行脚本在已有代码中重命名他们。

作为对反对使用的不安全生命周期函数的补充，我们新加入了一些新的生命周期函数：

  • getDerivedStateFromProps 作为一种更安全的方式，替代之前的 componentWillReceiveProps。

  • getSnapshotBeforeUpdate 用于安全的读取属性，比如在 DOM 更新前被使用。

    StrictMode Component

    StrictMode 是用于高亮在应用中的潜在问题的工具。就像 Fragment，StrictMode 不渲染任何可见的 UI。它会激活额外的检查和后代的警告。

    注意
    StriceMode 检查只会在开发模式下生效，而不会影响生产模式。

    尽管 StrictMode 不可能发现所有的问题（比如确定突变的类型），它也可以帮到你很多东西。如果你在 strict mode 下看到了警告，它们很可能在异步渲染的时候造成问题。

    在 16.3 版本中，StrictMode 可以帮助以下问题：

  • 判断组件是否有不安全的生命周期函数

  • 警告使用了之前的字符串 ref API 用法。

  • 检测一些难以预料的影响

    别的的功能会在以后的 React 版本中加入。
